# Sprint Retrospectives

## Why this exists

Retros were happening conversationally with no written record — useful in the moment, but nothing to point back to for the SPM deliverable, and no way to check whether "the one change" from a retro actually happened. This directory makes each sprint's retro a real artifact instead of relying on memory.

## Convention

- One file per sprint: `docs/retros/sprint-N.md` (e.g. `sprint-2.md`, `sprint-3.md`).
- Copy `TEMPLATE.md`, don't edit it in place.
- Filled out at the end of each sprint — the natural point to do this is right after the sprint's PPP/review meeting, since the team is already together and the sprint's outcomes are fresh.
- Whoever runs the retro that sprint fills it in and commits it directly to `main` (docs-only change, no PR required, but a PR is fine too if it's easier).

## Sections

- **Follow-up from last sprint's change** — closes the loop on the previous retro's one action item. This is the part that actually makes retros compound instead of resetting every sprint.
- **What went well / What didn't go well** — plain bullet points, specific enough to be useful later (ticket numbers, names, concrete events — not "communication could be better").
- **One change for next sprint** — exactly one. Forcing a single pick keeps it actionable instead of a wishlist that never gets checked.
